🔧 배포 확인 & 트러블슈팅

GLOSSARY
시스템 용어 설명
레포 (Repository)
코드와 파일을 관리하는 저장소. 한 프로젝트의 모든 파일·변경 이력이 한 곳에 묶여 있는 폴더 같은 개념. 현재 2개 레포 운영: gs_daily_trend_news_public_temp (트렌드/전략/실적/마감 리포트) + sono-competitor-crawler (경쟁사 모니터링 / 팔라티움 리포트).
파싱 (Parsing)
팔라티움에서 내려받은 원시 텍스트 파일(온북 데이터)을 프로그램이 읽을 수 있는 구조화된 데이터로 변환하는 과정. 엑셀에서 텍스트 나누기를 수백만 행에 대해 자동으로 실행하는 것과 같습니다.
Git
파일 변경 이력을 추적하는 버전 관리 도구. "v1, v2, 최종, 진짜최종" 파일을 여러 개 만드는 대신, 하나의 파일에서 변경 기록을 자동으로 쌓아가는 방식입니다.
커밋 (Commit)
변경사항을 Git에 저장하는 행위. "이 시점의 파일 상태를 기록해둬"라는 뜻이며, 메모와 함께 저장되어 나중에 어떤 수정이었는지 추적할 수 있습니다.
풀 (Pull)
원격 저장소(GitHub)에 있는 최신 변경사항을 내 컴퓨터로 가져오는 동작. 자동 파이프라인 등 다른 곳에서 올린 변경분을 로컬에 동기화합니다.
푸시 (Push)
내 컴퓨터에서 커밋한 변경사항을 GitHub에 올리는 동작. 이 작업을 해야 웹사이트(GitHub Pages)에 실제 반영됩니다.
샌드박스 (Sandbox)
AI가 코드를 실행하는 격리된 가상 환경. 파일 읽기/쓰기와 파이썬 실행은 가능하지만, 보안상 외부 네트워크(GitHub 등) 접속이 제한되어 git push는 로컬 터미널에서 실행해야 합니다.
체크포인트 방식
대용량 파싱(660만행 이상)을 한 번에 처리하면 시간 초과가 발생하여, 작업을 여러 단계(phase)로 나누어 각 단계 완료 시 중간 결과를 저장하고 이어서 처리하는 방식. 게임의 세이브 포인트와 같은 개념입니다.
스태시 (Stash)
작업 중이던 변경사항을 임시 서랍에 넣어두는 기능. 아직 커밋하지 않은 수정 내용을 잠깐 치워놓고 다른 작업(예: pull)을 먼저 할 때 사용합니다. stash pop은 서랍에서 다시 꺼내 적용하는 것. 주의: pop 시 구버전 파일이 최신 파일을 덮어쓸 수 있어 충돌이 발생할 수 있습니다.
워크트리 (Worktree)
하나의 레포에서 여러 브랜치를 동시에 작업할 수 있도록 별도 폴더를 만드는 기능. AI가 여러 작업을 병렬로 처리할 때 각각 독립된 워크트리에서 코드를 수정하고, 완료되면 메인(main)에 머지합니다. 작업 간 충돌을 방지하는 장점이 있지만, 머지 후 메인에서 재빌드를 해야 반영됩니다.
머지 (Merge)
별도 브랜치(워크트리)에서 작업한 변경사항을 메인 브랜치에 합치는 작업. 두 사람이 각자 수정한 문서를 하나로 합치는 것과 같습니다. 같은 부분을 다르게 수정한 경우 충돌(conflict)이 발생하며, 이를 수동으로 해결해야 합니다.
